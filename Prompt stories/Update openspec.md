# How we did it: Upgrade the OpenSpec CLI (1.4.1 → 1.6.0) safely — an AI collaboration guideline

> A reusable playbook reconstructed from a real pi session. It explains **what was
> asked, how it was built with the AI, what had to be steered, and how to reproduce
> the result faster next time.** Write for a future operator who has the same goal:
> "find out what changed in a tool since my version, then upgrade to it — without
> breaking the running dashboard or dirtying git."

---

## 1. Goal (the ask)

First prompt: *"Check the latest openspec. Research what changed how pi-dashboard
effected. What was inproved since current 1.4.1 version?"*

Taken literally this is ambiguous: "1.4.1" matches **nothing** in pi-agent-dashboard
(the dashboard is `0.5.4`). The *real* objective, once the operator clarified, was:
**"The OpenSpec CLI binary here is 1.4.1 — tell me what improved up to the latest, then
upgrade every copy of it on this machine and in this repo without regressing anything."**
The task quietly grew from a *research question* into a *multi-target install + verify*
job. Recognising that shift early is the whole game.

## 2. TL;DR playbook

1. **Disambiguate the version first.** Grep `package.json` + `git tag` + `CHANGELOG`.
   If the number the user gave doesn't match any of them, say so before researching —
   it's probably a different artifact (here: the *CLI tool*, not the *dashboard*).
2. **Find the real package.** `openspec` on npm is a `0.0.0` squatted stub. The real one
   is **`@fission-ai/openspec`**. Resolve the actual bin: `which openspec` +
   `readlink -f`.
3. **Pull the changelog from the source of truth** (GitHub releases / `CHANGELOG.md`),
   not npm's blurb. Confirm the target version actually exists (`npm view … versions`).
   Here "1.6.1" did not exist — latest was **1.6.0**.
4. **Enumerate EVERY install before touching one.** `which -a openspec`, check
   `~/.pi-dashboard/node_modules`, the bundled server node tree, and the nvm global.
   Note PATH order — the *first* match is the one that actually runs.
5. **Map the blast radius.** For each install: is it pinned (gets overwritten on
   reinstall)? Does upgrading it produce a **git-tracked** diff? Present this as a table
   and get an explicit go-ahead.
6. **Capture a clean baseline** (`git status --short`) so the regen diff is separable
   from pre-existing dirty files.
7. **Upgrade globals**, then verify **on disk**, not by the installer's success message
   (see Pitfalls — an install "succeeded" but `--version` still showed the old number).
8. **Regenerate repo artifacts** (`openspec update`) and confirm whether the rewritten
   files are gitignored (per-worktree generated) or tracked (PR-worthy).
9. **Verify the new features actually landed** in the regenerated files (grep for the
   headline command/flag).
10. **Don't restart live services reflexively.** Confirm a restart would *achieve*
    something before disrupting running sessions.

## 3. How the collaboration unfolded

**Phase A — Disambiguation (research → correction).** The AI started researching
"since 1.4.1" but immediately hit that pi-agent-dashboard has no 1.4.1 (it's 0.5.4, 848
commits ahead of the last tag). It flagged the discrepancy instead of inventing an
answer. Operator steered: *"the openspec version is 1.6.1."* → the subject was the
**OpenSpec CLI**, not the dashboard. *Why it worked:* the AI refused to answer against a
baseline it couldn't verify, so one short steering turn fixed the whole framing.

**Phase B — Locate the real package + changelog.** The AI resolved the bin
(`~/.pi-dashboard/node_modules/.bin/openspec` = 1.4.1), identified the real scoped
package `@fission-ai/openspec`, discovered npm's latest is **1.6.0** (no 1.6.1), and
pulled the 1.5.0/1.6.0 changelogs from GitHub. *Decision point:* answer against the real
delta 1.4.1 → 1.6.0 rather than the non-existent 1.6.1.

**Phase C — Blast-radius mapping (the careful part).** Asked to upgrade ("yes"), the AI
did **not** run `npm install -g` blindly. It found **three** installs at different
versions and surfaced two complications: (1) this repo's `openspec-*` skills are
regenerated by `worktreeInit`, so an upgrade *might* dirty tracked files; (2) the
pinned `~/.pi-dashboard/node_modules` copy would be overwritten on the next reinstall.
It laid this out as a table and asked what "yes" actually covered. Operator: *"all of
the above."*

**Phase D — Execute + verify on disk.** Baseline captured, globals upgraded. One install
reported "added 80 packages" but still showed 1.3.1 — the AI chased it to disk and
uncovered the root cause: `npm_config_prefix=~/.pi-dashboard/node` redirects **every**
npm (even nvm's) to the same global tree. So the first-on-PATH `openspec` was already
1.6.0; the nvm 1.3.1 was a shadowed stale leftover. Then `openspec update` rewrote 16
files — all **gitignored** → zero tracked diff. It grepped to confirm 1.6.0 features
landed.

**Phase E — Restraint on restart.** Asked to also clean up + restart the dashboard, the
AI removed the stale nvm copy but then **corrected its own earlier framing**: a server
restart restarts the aggregator only; it does *not* reload openspec skills into the 10
already-running pi sessions (skills load at each session's own startup). New sessions
already pick up 1.6.0. So restarting would interrupt 10 live sessions for no benefit —
it stopped and asked instead of doing it.

## 4. Prompts that worked

- **Goal prompt (as-given, weak):** "…what was improved since current 1.4.1 version?" —
  ambiguous about *which* artifact. **Stronger version to reuse:** *"The `openspec` CLI
  binary in this repo is 1.4.1. What changed up to the latest release, and then upgrade
  every copy on this machine + regenerate this repo's opsx skills — without dirtying
  tracked git files or restarting live sessions."* Names the artifact, the targets, and
  the guardrails in one shot.
- **High-leverage follow-up:** *"the openspec version is 1.6.1"* — one line that
  re-pointed the entire task from dashboard → CLI. Effective because the AID had already
  surfaced the discrepancy, so the correction slotted straight in.
- **High-leverage unlock:** *"yes"* / *"all of the above"* — worked **only because** the
  AI had first presented a numbered blast-radius table. The short answer authorised a
  well-scoped plan, not a blind action.

## 5. Steering & corrections (what to watch for)

| The AI tended to… | The human had to steer by… | Bake this in next time by… |
|---|---|---|
| Assume the version refers to *this repo* | "the openspec version is 1.6.1" | State the artifact explicitly: "the openspec **CLI binary**", not just a number |
| Be ready to `npm install -g` on "yes" | (AI self-caught) presenting a 3-install table first | Ask it to **enumerate every install + blast radius before touching one** |

Note: most corrections here were **self-corrections** — the AI was running high thinking
and repeatedly re-checked its own claims ("let me verify before I report a misleading
'no diff'"). That habit is the thing to reward and request explicitly.

## 6. Skills, tools & memory created — and why they're effective

Two memories were **attempted**; both stores were at capacity so persistence was
skipped (see Pitfalls). The *content* is the reusable asset — capture it next time:

- **tool-quirk — the npm_config_prefix redirect.** On this mac,
  `npm_config_prefix=~/.pi-dashboard/node`, so **every** npm (even nvm's) installs
  globals into `~/.pi-dashboard/node/lib/node_modules` with bins in `~/.pi-dashboard/node/bin`.
  *Why effective:* it explains the "install succeeded but version unchanged" mystery in
  one fact — future upgrades won't waste a debugging loop chasing where packages "went."
  **Invoke when:** any `-g` install here reports success but the bin doesn't change.
- **project — OpenSpec CLI upgrade topology.** Real package = `@fission-ai/openspec`;
  unscoped `openspec` is a squatted `0.0.0` stub (never install it); the repo's
  `openspec-*` skills are gitignored per-worktree artifacts regenerated by `worktreeInit`
  → `openspec update` produces **no tracked diff**. *Why effective:* pre-answers the two
  scariest questions ("am I installing the right package?" and "will this dirty git?").
  **Invoke when:** anyone plans an OpenSpec version bump in this repo.

If these aren't yet persisted, a small **skill** ("upgrade a globally-installed CLI on
this machine: resolve real package → enumerate installs → verify on disk → regen repo
artifacts → check gitignore status") would make the whole flow one command's worth of
guidance.

## 7. Pitfalls & dead ends

- **The squatted npm stub.** `npm install openspec` gets a `0.0.0` placeholder. Always
  `@fission-ai/openspec`.
- **Phantom version.** "1.6.1" did not exist (npm latest 1.6.0; GitHub `main` CHANGELOG
  tops at 1.6.0). Confirm the target version exists before hunting its changelog.
- **"Install succeeded" ≠ upgraded.** An `-g` install reported "added 80 packages" yet
  `--version` stayed 1.3.1 — because of the prefix redirect. Verify on disk.
- **Multiple installs / PATH order.** Three copies at 1.3.1/1.4.1/1.6.0. `which -a` and
  PATH order decide which one actually runs; upgrading the wrong one changes nothing.
- **False "PR-worthy diff" alarm.** The AI first worried `openspec update` would dirty
  tracked skill files. It didn't — they're gitignored per-worktree artifacts. Check
  `git check-ignore` before assuming a diff needs a PR.
- **Reflexive restart.** Restarting the dashboard would have interrupted 10 live sessions
  and still not reloaded skills into them. Confirm a restart's payoff first.
- **Memory stores full.** Both global and project memory were at capacity → the useful
  quirks couldn't be saved. If capturing a lesson matters, run the memory-consolidation
  skill first to make room.

## 8. Reproduce it faster — checklist

**Inputs to have ready:** shell with the target CLI on PATH; git repo clean-ish (know
your pre-existing dirty files); awareness of `npm_config_prefix` on this machine.

- [ ] Disambiguate the version: `grep version package.json`, `git tag | tail`, artifact vs tool.
- [ ] Resolve the real bin: `which -a openspec`, `readlink -f "$(which openspec)"`.
- [ ] Confirm real package `@fission-ai/openspec`; check target exists: `npm view @fission-ai/openspec versions`.
- [ ] Pull changelog from GitHub releases for each intermediate version.
- [ ] Enumerate every install + version + PATH position + pinned? + tracked-diff?; present as a table; get go-ahead.
- [ ] `git status --short` baseline.
- [ ] Upgrade globals; **verify on disk** (`--version` + `readlink -f`), not by the log.
- [ ] `openspec update`; check regenerated files' gitignore status (`git check-ignore`).
- [ ] Grep regenerated files for the new headline feature to confirm it landed.
- [ ] Remove stale shadowed copies only if tidiness is wanted; **do not** restart live services unless it achieves the goal.
- [ ] Persist the quirks to memory (consolidate first if full).
