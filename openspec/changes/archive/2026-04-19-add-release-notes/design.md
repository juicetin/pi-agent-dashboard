## Context

The dashboard has shipped 10 tags (`v0.2.0` → `v0.2.9`) with no `CHANGELOG.md` in the repo. Releases on GitHub are currently generated by `softprops/action-gh-release@v2` with `generate_release_notes: true`, which produces a flat GitHub-autogenerated "What's Changed" list straight from commit messages — noisy, unstructured, mixes CI fixes with real features. There is no documented release process and no single place a reader can find "what changed in v0.3.0."

Commit history is already mostly Conventional-Commits-shaped (`feat:`, `fix:`, `refactor:`, `docs:`, `chore:`, with occasional scopes), so a manual-but-disciplined workflow is achievable without adopting a generator tool.

Stakeholders:
- **End users** reading GitHub Releases / in-app update prompts → want a short, grouped, feature-level story.
- **Developers / contributors** reading `CHANGELOG.md` → want the complete list, including dev-facing refactors.
- **Release author** (whoever cuts the tag) → wants a low-friction workflow that won't break CI.

## Goals / Non-Goals

**Goals:**
- A single source of truth — `CHANGELOG.md` — that documents every release going forward, in [Keep a Changelog 1.1.0](https://keepachangelog.com/en/1.1.0/) format.
- A rich v0.3.0 entry that communicates the substantive work shipped since v0.2.9 (33 commits).
- A collapsed backfill entry for v0.2.0–v0.2.9 that acknowledges the releases without pretending they had user-facing feature content.
- GitHub Release body derived from the matching CHANGELOG section, with graceful fallback to existing behavior if extraction fails.
- A written release process in `docs/release-process.md` so future releases don't regress.

**Non-Goals:**
- Automated changelog generation (`release-please`, `changesets`, `conventional-changelog`). Explicitly deferred — current volume doesn't justify the tooling cost, and manual curation yields better user-facing prose.
- Enforcing Conventional Commits via CI lint (`commitlint`, husky hooks). Convention + code review only.
- In-app "What's New" dialog surfaced after upgrade. Separate future change.
- Per-version rich rewrites of v0.2.0–v0.2.8 — content isn't there.
- Semantic-version policy changes. We continue SemVer; this change is purely about communication.

## Decisions

### 1. Format: Keep a Changelog 1.1.0

**Chosen:** [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) with sections `Added / Changed / Deprecated / Removed / Fixed / Security` per version, plus a top `## [Unreleased]` section.

**Alternatives considered:**
- *Conventional-Commits grouping* (`### Features` / `### Bug Fixes`) — auto-generatable but poor end-user copy; commit subjects are optimized for reviewers, not users.
- *Free-form narrative* — readable but inconsistent over time; no stable anchor for automation.

**Rationale:** Keep a Changelog is the closest thing to an industry default, parses trivially (headings are predictable), and separates user-visible sections cleanly. It also has an explicit `Unreleased` convention that matches how we want PRs to contribute.

### 2. Versioning anchors match git tags verbatim

Each versioned section heading is `## [0.3.0] - YYYY-MM-DD` and the tag is `v0.3.0`. The single-character `v` prefix is present on tags but absent from CHANGELOG headings (KaC convention). Release-body extraction strips the leading `v` from `GITHUB_REF_NAME` before matching.

### 3. Backfill v0.2.0–v0.2.9 as one collapsed entry

**Chosen:** One section `## [0.2.0 – 0.2.9] - 2026-04-13 – 2026-04-16` summarizing "initial public releases — installer and cross-platform CI hardening," with 4–6 bullets covering: npm package rename to `@blackbelt-technology/pi-agent-dashboard`, Windows ZIP + portable installers, arm64 artifacts, AppImage (`libfuse2`), Docker build pipeline.

**Alternatives considered:**
- *One entry per version* — honest but wastes reader attention; most entries would read "fix CI".
- *Skip backfill entirely, start at v0.3.0* — leaves a confusing gap for anyone searching for v0.2.x.

**Rationale:** Users who land on `CHANGELOG.md` can see the project existed before v0.3.0 without being drowned in CI fixes. The range heading is a mild KaC deviation but is clearly signaled.

### 4. `Unreleased` section is mandatory and always present

Top of the file, immediately under the title, always contains:

```markdown
## [Unreleased]

### Added
### Changed
### Fixed
```

Empty subsections are allowed. PRs that ship user-visible behavior append a bullet under the right subsection. At release time, `Unreleased` heading is renamed to the versioned heading with a date, and a fresh empty `Unreleased` is inserted above.

### 5. GitHub Release body: extracted from CHANGELOG, with fallback

New workflow step before `softprops/action-gh-release@v2` runs a small script (inline Node or `awk`) that:

1. Reads `CHANGELOG.md`.
2. Strips the leading `v` from `${GITHUB_REF_NAME}` → target version string (e.g., `0.3.0`).
3. Finds the heading line matching `^## \[<version>\]` (exact match).
4. Captures everything up to the next `^## \[` heading.
5. Writes the captured block to `release-notes.md`.
6. Passes it to the release action via `body_path: release-notes.md`.

If extraction fails (no matching heading, empty body, script error), the step writes a one-line fallback note (e.g., "See `CHANGELOG.md` in the repo for full notes.") and the release action falls back to `generate_release_notes: true`.

**Alternatives considered:**
- *Set `body` inline in YAML* — would require regenerating the workflow every release. Dead end.
- *`release-drafter`* — another tool; same complaint as release-please.
- *Leave `generate_release_notes: true` as-is* — fails the whole goal.

### 6. `docs/release-process.md` as the canonical how-to

Single-page doc covering: commit conventions (recap), the `Unreleased` → versioned promotion, version bump (`npm version <x> --workspaces --include-workspace-root`), tag + push, what the CI workflow does automatically, and a manual fallback for editing the GitHub Release if CHANGELOG extraction mis-rendered.

### 7. No CI enforcement (yet)

PRs can land without a CHANGELOG entry. The release-process doc sets the expectation that the release author reviews `Unreleased` at tag time and fills gaps. Rationale: keep the bar for contributors low; the release author is already doing a review pass.

## Risks / Trade-offs

- **Risk:** Contributors forget to update `Unreleased`. **Mitigation:** release author does a sweep of `git log <last-tag>..HEAD` at tag time and back-fills; process doc makes this explicit.
- **Risk:** CHANGELOG extraction regex breaks on an edge case (e.g., a bullet that starts with `## [` inside a code block). **Mitigation:** extraction script requires `^## \[` at column 0 and runs in a step that only logs a warning on failure — release still publishes via the fallback.
- **Risk:** The range heading `## [0.2.0 – 0.2.9]` confuses parsers expecting a single version. **Mitigation:** it never needs to match — no tag `v0.2.0` → `v0.2.9` will ever be pushed again. Extraction is only run on future tags.
- **Trade-off:** Manual curation costs release-author time per release. Accepted — at current release cadence (~1/week at peak, less now) the marginal effort is small and the output quality is much higher than automated.
- **Trade-off:** No `CHANGELOG.md` lint means drift is possible. Accepted initially; can be tightened later with a simple CI check if needed.

## Migration Plan

1. Land this change's files (CHANGELOG.md, docs/release-process.md, publish.yml edit, README/AGENTS links) on `main`.
2. No tag is cut as part of this change — the first release that uses the new flow will be `v0.3.0`, cut as a separate commit after this change archives.
3. Rollback: revert the PR. `publish.yml` fallback means even a broken extraction step won't block a release.
