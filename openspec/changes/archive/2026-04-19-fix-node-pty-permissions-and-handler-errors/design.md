## Context

The dashboard server uses `node-pty` to spawn PTYs for the terminal feature. `node-pty` ships prebuilt native artifacts under `node_modules/node-pty/prebuilds/<platform>/`:

- `pty.node` — the native addon
- `spawn-helper` — a small executable invoked via `posix_spawnp` on macOS/Linux

In certain environments, `npm` unpacks `spawn-helper` without the execute bit (`-rw-r--r--`). Calling `pty.spawn(...)` then fails with `posix_spawnp failed.` An existing postinstall script (`packages/server/scripts/fix-pty-permissions.cjs`) was written to chmod the helper, but it computes the path as:

```js
path.join(__dirname, "..", "node_modules", "node-pty", "prebuilds")
```

That resolves to `packages/server/node_modules/node-pty/prebuilds/`. In this monorepo, npm workspaces **hoist** `node-pty` to the root `node_modules/`, so the script sees `ENOENT`, its outer `try/catch {}` swallows it, and nothing is chmod-ed.

The failure is then hidden a second time by `packages/server/src/browser-gateway.ts`, which wraps the entire browser-message `switch` in:

```ts
try { switch (msg.type) { ... } } catch { /* Ignore malformed messages */ }
```

Any exception thrown inside a handler (e.g. `terminalManager.spawn` throwing `posix_spawnp failed.`) is swallowed with zero logging, so the server looks healthy while the feature appears "dead" to the user.

Both issues are independently real and each has a clean, local fix. This change addresses them together because they were the cause-and-obscurer of a single user-visible bug.

## Goals / Non-Goals

**Goals:**
- `npm install` in a fresh clone leaves `node-pty` prebuild `spawn-helper` files executable on macOS/Linux, regardless of where npm places `node-pty` (hoisted root or workspace-local).
- `browser-gateway.ts` logs any exception thrown by an individual message handler instead of silently swallowing it. The narrow case of malformed JSON frames still does not spam logs.
- Regression coverage exists so a future `node-pty` upgrade or npm layout change that reintroduces either bug will fail CI rather than reach users.

**Non-Goals:**
- Not touching `node-pty` itself or pinning a specific version.
- Not redesigning error reporting in the browser gateway beyond logging; structured error-to-client responses are out of scope (can be a follow-up).
- Not altering the Electron bundled-server path (`packages/electron/resources/server/node_modules/`); the same fix naturally applies whenever that tree is regenerated by `bundle-server.sh`.
- Not changing terminal UX, xterm.js wiring, or `terminal-manager.ts` behavior.

## Decisions

### D1. Locate `node-pty` via `require.resolve`, not a hardcoded path.

Rewrite `fix-pty-permissions.cjs` to do:

```js
const ptyPkg = require.resolve("node-pty/package.json");
const prebuildsDir = path.join(path.dirname(ptyPkg), "prebuilds");
```

Then walk `prebuildsDir`, chmod every `*/spawn-helper` (and, defensively, `*/pty.node`) to `0o755`.

**Alternative considered:** walk up from `__dirname` manually looking for a `node_modules/node-pty` directory. Rejected — `require.resolve` is the canonical Node API for this and handles pnpm/yarn-berry layouts too.

**Alternative considered:** run `npm rebuild node-pty`. Rejected — much heavier, requires a working toolchain, and the problem isn't a build problem, it's a permission bit on a shipped prebuild.

### D2. Run the script at the repo-root `postinstall`, not only under `packages/server`.

Add `postinstall` to the root `package.json` pointing at the (single) script. The existing `packages/server/package.json` `postinstall` can stay (idempotent) or be removed; we keep it for safety so a `npm install` executed inside `packages/server/` alone still fixes permissions.

**Rationale:** Workspaces hoist `node-pty` up; the root install is the moment when it's definitely on disk. Running there guarantees the fix applies once per `npm install`, regardless of which workspace initiated it.

### D3. Narrow the `browser-gateway.ts` catch-all to JSON parsing only.

Shape:

```ts
ws.on("message", (raw) => {
  let msg: BrowserToServerMessage;
  try {
    msg = JSON.parse(raw.toString());
  } catch {
    return; // malformed frame, silently drop (existing behavior)
  }
  try {
    switch (msg.type) { ... }
  } catch (err) {
    console.error(
      `[browser-gw] handler error type=${(msg as { type?: string })?.type ?? "unknown"}:`,
      err,
    );
  }
});
```

**Rationale:** Preserves the "ignore garbage frames" behavior (which is legitimate defense), but promotes genuine handler bugs to visible log lines. Uses `console.error` to match existing gateway logging style (`[browser-gw] browser client connected ...` uses `console.error`).

**Alternative considered:** also send an `error` frame back to the originating WebSocket so the UI can toast. Rejected for this change — most handlers don't carry a correlation id today, so the frontend would not know which user action to blame. A structured error-response protocol is a separate proposal.

### D4. Regression tests.

Two tiny tests, both cheap and deterministic:

1. **`fix-pty-permissions` test** (node, synchronous):
   - Resolve `node-pty/package.json`, locate `prebuilds/<current platform>/spawn-helper` (skip on Windows).
   - Assert the file exists and that `fs.statSync(...).mode & 0o111` is non-zero (some execute bit set).

2. **`browser-gateway` handler error test** (vitest):
   - Construct a minimal context with a `terminalManager` stub whose `spawn` throws.
   - Send a `create_terminal` message through the gateway's `onMessage` path.
   - Assert `console.error` was called with a message including `[browser-gw] handler error` and `type=create_terminal`.

These tests document the invariants and pin the behavior.

## Risks / Trade-offs

- **[Risk] Previously swallowed exceptions become visible log noise after restart.** → Mitigation: that's the point; any new noise is a real bug that was already happening. If noise proves excessive we can switch specific handlers to structured error responses.
- **[Risk] `require.resolve("node-pty/package.json")` throws if `node-pty` is not installed (e.g. a workspace that doesn't depend on it runs the root postinstall).** → Mitigation: wrap in try/catch and no-op, matching the current defensive style.
- **[Risk] Some package managers (pnpm isolated mode) place `node-pty` in yet another layout.** → Mitigation: `require.resolve` handles pnpm/yarn-berry too, and the test runs under the actual install so any regression surfaces in CI.
- **[Trade-off] The root `postinstall` runs on every `npm install`, even when `node-pty` hasn't changed.** → Acceptable: a handful of `chmod` syscalls, sub-millisecond.

## Migration Plan

1. Land the proposal + code changes in one commit sequence.
2. On merge, CI runs `npm install` → postinstall fires → spawn-helper is executable → tests pass.
3. Existing broken installs: a single `npm install` (or manual `chmod +x node_modules/node-pty/prebuilds/<platform>/spawn-helper`) resolves it. No data migration, no API change.
4. Rollback: revert the commit. No persistent state is introduced.

## Open Questions

- None at this time. If a follow-up adds structured error responses to the browser WebSocket protocol, the handler error logging added here becomes the natural place to also emit those.
