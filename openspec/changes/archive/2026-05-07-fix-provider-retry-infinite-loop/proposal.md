## Why

When an LLM provider returns a persistent rate-limit / quota / usage-limit error (e.g. Codex `usage_limit_reached`, Anthropic `overloaded_error` that doesn't clear), the dashboard wedges for several minutes with the spinner stuck and the **Stop button does nothing** â€” the only recovery is killing the pi process. Two compounding causes:

1. **Three nested retry layers** (pi-ai HTTP retry: 3Ă— backoff to 8s; pi-coding-agent `_handleRetryableError`: 3Ă— backoff to 16s; both regex-match the same `/rate.?limit|429|too many requests/i` error text). A single 429 from the provider produces ~30+ s of unobservable retrying inside pi-coding-agent on top of pi-ai's own ~15 s â€” and quota errors are *not* transient so all retries fail.
2. **The dashboard is invisible to the retry phase.** pi-coding-agent emits `auto_retry_start` / `auto_retry_end` events, but `event-reducer.ts` doesn't handle them. The bridge strips the failed assistant message before retrying (per pi-coding-agent's contract), so `agent_end` arrives with no `stopReason: "error"` between retries either. From the dashboard's perspective the session just sits in `streaming` state for minutes.
3. **Stop is racy.** `AgentSession.abort()` clears `_retryAbortController`, but between `_handleRetryableError` resolving and the next `agent.continue()` call there is a synchronous window where a fresh AbortController gets installed, so an Abort that arrives in that window is dropped.

Fix the user-visible symptom (loop + dead Stop) at the dashboard layer where we control: surface the retry phase, prevent retry on known-non-retryable usage-limit errors, and make Stop definitively cancel.

**Architectural constraint discovered during implementation**: pi's `ExtensionAPI.on(...)` enumeration does NOT include `auto_retry_start` / `auto_retry_end` events. They fire via `AgentSession._emit â†’ _eventListeners` (the embedded-SDK `subscribe()` API used by interactive-mode and rpc-mode), and `AgentSession._emitExtensionEvent` has no fall-through case for them. Verified against `agent-session.js` (lines 376â€“452, 1964/1986/324), `types.d.ts` ExtensionAPI overloads, and EventBus emitters â€” see [pi-mono discussion #2073](https://github.com/badlogic/pi-mono/discussions/2073) for the user-facing version of the same gap. **Therefore the bridge synthesizes the events itself** by observing `message_end` (assistant `stopReason: "error"` with retryable `errorMessage`) and `agent_end`, both of which DO arrive via the extension API.

## What Changes

- **Bridge synthesizes auto-retry events from observed pi events**: a new pure helper `retry-tracker.ts` consumes the bridge's existing `message_end` and `agent_end` event flow. When an assistant `message_end` arrives with `stopReason: "error"` AND `errorMessage` matches pi-coding-agent's retryable regex (overloaded / rate.?limit / 429 / 5xx / network / timeout / fetch failed / terminated / retry delay), the tracker emits a synthesized `auto_retry_start` event for that session. When the next non-error `message_end` (success), `agent_end`, or user abort arrives, it emits `auto_retry_end`. `delayMs` and `maxAttempts` are unknowable (pi doesn't tell us its retry settings); we send `delayMs: -1` and `maxAttempts: -1` as sentinels that the UI renders as an indeterminate spinner instead of a countdown.
- **Reducer + retryState**: `event-reducer.ts` SHALL handle synthesized `auto_retry_start` / `auto_retry_end`, exposing transient `retryState` on `SessionState` (`{ attempt, maxAttempts, delayMs, reason, startedAt }`). `agent_start` / `agent_end` defensively clear it.
- **Retry banner**: `ChatView` SHALL render a transient "Rate-limited â€” retryingâ€¦" banner above the input while `retryState` is set. When `delayMs > 0` AND `maxAttempts > 0` the banner SHALL show "retry N of M in Xs" with a countdown; otherwise it SHALL show an indeterminate "retryingâ€¦" message. The banner SHALL include a **Stop retrying** button that sends `abort` (same wire as Stop). Distinct from the existing red `ErrorBanner`, which only shows after retries are exhausted.
- **Session card retry indicator**: while `retryState` is set, the session card SHALL show an amber pulsing dot.
- **Reliable Abort during retry**: bridge `command-handler.ts` SHALL call `cachedCtx.abort()` synchronously, IMMEDIATELY send a synthesized `auto_retry_end { success: false, finalError: "Aborted by user" }` (so the UI clears optimistically even if pi-coding-agent's actual `auto_retry_end` is delayed), AND start a **persistent-abort scheduler** that re-invokes `cachedCtx.abort()` every 200ms for up to 2s OR until `cachedCtx.isIdle()` returns true â€” closing the race window where pi-coding-agent unsets `_retryAbortController` between attempts.
- **Non-retryable usage-limit detection**: pure helper `usage-limit-orderer.ts` watches the bridge's `agent_end` flow. When the terminal `errorMessage` matches `/usage[_ ]limit[_ ]reached|usage_not_included|quota[_ ]exceeded|monthly limit|hourly limit|reset after \d+[hms]/i` AND a synthesized retry was in flight for that session, it forwards `auto_retry_end { success: false, finalError: <error> }` BEFORE the `agent_end` so the UI clears the retry banner before showing the error banner (no transient frame with both visible).
- **Stop button visibility extended**: `CommandInput` Stop / Force-Stop predicate counts `retrying === true` as "still working", since `sessionStatus` may briefly read `idle` between retries.

## Capabilities

### New Capabilities
- `provider-retry-state`: Tracks and renders in-flight LLM provider retries (auto_retry_start/end events, retryState session field, retry banner UI, session-card amber dot).

### Modified Capabilities
- `play-stop-controls`: Stop button SHALL also abort active retry sleeps and SHALL escalate to Force Stop after a 3 s grace period when the session is in retry state. Add scenario covering Stop pressed during retry phase.
- `error-detection`: Add scenario for `agent_end` errorMessage matching usage-limit / quota patterns producing immediate terminal state without further retries; add scenario clarifying that `lastError` is set ONLY when pi-coding-agent gives up retrying (i.e. retries are exhausted), not on every transient `auto_retry_start`.

## Impact

- **Code**: `packages/client/src/lib/event-reducer.ts` (+`retryState` field, two new event arms), `packages/client/src/components/ChatView.tsx` (retry banner integration), new `packages/client/src/components/RetryBanner.tsx`, `packages/client/src/components/CommandInput.tsx` (Stop visibility on `retrying`), session card amber dot, new `packages/extension/src/retry-tracker.ts` (pure helper: synthesizes `auto_retry_*` from observed message_end/agent_end), new `packages/extension/src/usage-limit-orderer.ts` (pure helper: synthesizes pre-agent_end retry-end on usage-limit match), `packages/extension/src/command-handler.ts` (synthesize abort + persistent-abort scheduler), `packages/extension/src/bridge.ts` (wire trackers into the message_end / agent_end forwarding paths).
- **Protocol**: No new wire types. The bridge only emits `event_forward` messages whose `eventType` is one of `auto_retry_start` / `auto_retry_end`. Reducer already shape-tolerates them.
- **Specs**: New `provider-retry-state`; deltas to `play-stop-controls` and `error-detection`.
- **Tests**: Pure-helper tests for `retry-tracker` and `usage-limit-orderer`, reducer tests for `auto_retry_*`, `RetryBanner` render test, `CommandInput` retrying-state test, `command-handler` abort-synthesis test.
- **Out of scope** (explicitly): adding `pi.on("auto_retry_*")` to pi's extension API (upstream work, tracked at [badlogic/pi-mono#2073](https://github.com/badlogic/pi-mono/discussions/2073)); changing pi-coding-agent's retry decisions; tuning pi-ai's per-provider HTTP retries; building a faux-provider integration harness.
- **No breaking changes**: existing `ErrorBanner` / `lastError` behavior unchanged for terminal errors. Existing event types remain unchanged.
