KB folder row — index feedback states

Change fix-kb-index-feedback. The row that shows KB · … → inside a folder / worktree card. Same five-state model as shipped; this change makes three previously-unreachable or silent states actually surface.

What the fix changes: the indexing spinner becomes reachable from Index now (route returns 202, client polls), a rejected trigger shows a real error instead of a dead button, and a transient poll blip no longer kills the spinner.

States the fix makes correct

▸os-attribute-openspec-poll-eventloop-stallsworktree
KB · INDEXING… 312 FILES → ↻

Indexing now reachable

After Index now, the route replies 202 {status:"running"} and the client polls /stats; the animated ↻ + live file count render for the whole walk.

Before: POST blocked until the walk finished, so the poll never saw indexing:true — the card sat frozen on NOT INDEXED, then jumped to populated. No spinner ever.

▸os-attribute-openspec-poll-eventloop-stallsworktree
KB · INDEX FAILED →
403 cwd not allowed — folder left the known set (session ended / unpinned)

Trigger rejected was silent

The POST /reindex itself is rejected (403 / 500 / transport) so no job starts. A distinct reindexError surfaces a red failed state + reason + Retry.

Before: FolderKbSection discarded the hook's error — the click did nothing visible, indistinguishable from a dead button.

▸os-attribute-openspec-poll-eventloop-stallsworktree
KB · INDEXING… 640 FILES → ↻
one /stats poll hiccuped — tolerated, still polling

Transient poll blip spinner survives

A single failed /stats poll while the walk runs no longer flips the row to failed. Polling is resilient (tolerates a bounded run of misses), so it recovers to the true terminal state.

Naive fix danger: an unconditional error render would show index failed here while the server is still happily walking — the doubt review caught this.
