## Why

The per-folder KB row (`FolderKbSection`, shipped by `add-kb-folder-slot`) offers an **Index now** button on a not-indexed folder, but clicking it gives the user no observable feedback and — when it fails — no signal at all. Investigation of a real worktree (`os-attribute-openspec-poll-eventloop-stalls`) traced two distinct client-side defects. The server and indexer are healthy: run directly, `indexSource` walks 974 files → 13,757 chunks, and `POST /api/kb/reindex` returns `200 {changed:974, chunks:13757}`. The failure is entirely in the client feedback loop.

**Bug 1 — the `indexing…` spinner never shows.** The `POST /api/kb/reindex` route does `return await promise` — it blocks the HTTP response until the whole walk finishes. The client `reindexKb(cwd)` awaits that fetch, then calls `refetch()`. So `useKbStats` never polls `/stats` *during* the walk and never observes `indexing: true`. The card sits frozen on `KB · NOT INDEXED` for the entire (multi-second, thousands-of-files) first index, then jumps straight to `populated`. The archived spec already promises "Running job shows progress → animated indicator" (Requirement: KB row reflects index state), but the blocking route makes that scenario unreachable from the button. The `PUT /api/kb/config` reindex path is already fire-and-forget and *does* animate — only the direct button blocks. Design inconsistency.

**Bug 2 — reindex errors are swallowed.** `FolderKbSection` destructures only `{ stats, reindex }` from `useKbStats` and never renders `error`. The hook faithfully sets `error` on any reindex failure (403 `cwd not allowed`, 500, a dropped/timed-out long POST on a large first index), but the component discards it. A click that silently errors is visually indistinguishable from a click that does nothing. Combined with Bug 1's missing spinner, the user sees a dead button. The empty schema-only DB observed on the worktree is the tell: no successful reindex had ever landed, yet the UI showed no error.

Turning a silent, feedback-free button into a visibly-progressing, failure-surfacing one is the whole point of the KB row's five-state design; today two of those states (`indexing`, client-side `error`) are unreachable via the primary action.

## What Changes

- **Make the `Index now` / reindex action non-blocking** so the `indexing` state becomes observable. The `POST /api/kb/reindex` route starts the job via the existing per-cwd registry and responds immediately (`202 { status: "running", jobId }`) instead of `await`-ing the walk to completion. The client, on a `running` response, calls `refetch()` so `useKbStats` begins polling `/stats`, observes `indexing: true`, and renders the animated indicator until the job settles — exactly the flow the config-reindex path already uses.
- **Surface reindex errors in the row — without eating the spinner.** `useKbStats` today collapses a *trigger* failure (the reindex POST rejected → no job started) and a *transient poll* failure (a `/stats` hiccup mid-walk) into one `error` and stops polling on either. Split them: expose a distinct `reindexError` for a rejected trigger (renders failed + `Retry` immediately), and make `/stats` polling **resilient** (tolerate a bounded run of transient misses instead of `clearPoll()`-ing on the first) so a live walk keeps its spinner and the true terminal state (`jobStatus:"error"` or `populated`) is still observed. `FolderKbSection` renders failed + `Retry` on `reindexError` OR server `jobStatus:"error"`, and keeps the spinner while `indexing:true`.
- **Make the walk non-blocking so the spinner is actually observable.** Discovered during implementation: the reindex walk (`indexSource`) is *fully synchronous* (`readdirSync` + `better-sqlite3` sync inserts), so even behind a `202` it pins the single Node thread for the whole walk — a concurrent `/stats` poll can't be *answered*, so `indexing:true` is never seen and no spinner renders (the exact case being fixed). `indexSource` becomes **async with batched commits**: it commits + `await setImmediate` every N files, yielding the event loop and releasing the WAL write lock so `/stats` is served mid-walk (and reports live progress). A `PRAGMA busy_timeout` makes a concurrent reader wait for a batch rather than fail `SQLITE_BUSY`. This trades whole-walk atomicity for responsiveness (a mid-walk throw leaves earlier batches committed; reindex is idempotent so a re-run completes it).
- **No change to config layering, cwd validation, or the DB schema.** The client fixes wire hook values through one component; the indexer change is transaction batching only (same rows, same graph).

## Capabilities

### Modified Capabilities
- `kb-folder-slot`: the **KB reindex route** requirement changes from a blocking (`return await promise`) response to a non-blocking start (respond `202 { status: "running" }` once the job is registered; the walk continues in-process and completion is observed via `/stats` polling). The **KB row reflects index state** requirement gains an explicit scenario that a client-initiated reindex which rejects (transport/HTTP error, not only server `jobStatus`) surfaces a visible failed state with `Retry`, closing the swallow-the-error gap.

## Impact

- **Server**: `packages/kb-plugin/src/server/kb-routes.ts` — the `POST /api/kb/reindex` handler stops `await`-ing the job promise; it returns `202 { status: "running", jobId }` after `registry.start()` and lets the walk run to completion in-process (attaching a `.catch` so an unhandled rejection is logged, mirroring the `PUT /config` fire-and-forget path). `reindexAll` becomes `async` (awaits `indexSource`). The coalesced-in-flight `202` branch is unchanged. Stats/config routes untouched.
- **Layer-1 engine (expanded scope)**: `packages/kb/src/indexer.ts` — `indexSource` becomes `async`, committing + yielding (`setImmediate`) every `YIELD_EVERY` files. `packages/kb/src/sqlite-store.ts` — `PRAGMA busy_timeout=5000`. Ripples to every caller (now `await`): `kb/src/cli.ts` `runIndex`, `kb-extension/src/reindex.ts` `reindexNow` (+ its debounce/`kb_search` callers in `extension.ts`), `kb/verify.ts`, and the `kb` + `kb-extension` test suites.
- **Client**: `packages/kb-plugin/src/client/useKbStats.ts` — add a distinct `reindexError` (trigger reject) separate from stats-poll errors; make the `/stats` poll tolerate a bounded run of transient failures instead of `clearPoll()`-ing on the first miss. `packages/kb-plugin/src/client/FolderKbSection.tsx` — render failed + `Retry` on `reindexError` or server `jobStatus:"error"`; keep the spinner while `indexing:true`.
- **Docs/types drift the change carries**: `kb-routes.ts` route JSDoc, `packages/kb-plugin/README.md`, and `KbReindexResult` in `shared/kb-plugin-types.ts` document the old `200 { changed, chunks }` POST body — update to the `202 { status:"running", jobId }` success shape. The route's now-dead `try/catch`→`500` block is removed (surgical-changes).
- **Tests**: `packages/kb-plugin/src/server/__tests__` — reindex route now asserts an immediate `202 { status: "running" }` and eventual `/stats` `chunks > 0`, replacing the synchronous-`200`-body assertion. `packages/kb-plugin/src/client/__tests__/*.test.tsx` — assert the spinner renders while `indexing: true` after a click, and that a rejected reindex renders the error/`Retry` state.
- **Spec parity restored**: the archived `KB row reflects index state` "Running job shows progress" scenario becomes actually reachable from the primary action; no scenario is deleted.
- **Out of scope**:
  - Live file-count *progress* during the walk (the route reports final counts; per-file streaming/WS progress is a separate enhancement — the spinner + polled chunk count is the v1 feedback).
  - **Cross-process durability of a reindex job** — a server restart mid-walk loses the in-memory job (row falls back to its committed count; a re-click recovers, since reindex is idempotent). Documented trade-off, not fixed here.
  - Changing cwd validation, the job registry's coalescing/retention, or the `dox-staleness.json` stale path.
  - The in-session `kb-extension` reindex triggers (untouched).
  - Any redesign of the settings page or the five-state visual language beyond wiring the client-error state.
