“Initialize” — one button, two meanings → split into two

The sidebar shows a single amber Initialize for both scaffold a new pi project and run the repo worktree-init hook. And an already-configured project with no hook is wrongly offered scaffold. This change splits the control by init-status state.

Before — today

new-idea ~/scratch/new-idea ① bare dir
my-feature ~/proj/wt/my-feature ② hook, unprovisioned
pi-agent-dashboard ~/proj/pi-agent-dashboard ③ configured, no hook

✕ Identical amber pill for ① and ②; you can't tell scaffold from “run repo code”. ✕ ③ is already a pi project yet still offers scaffold (dashed red).

After — this change

new-idea ~/scratch/new-idea ① bare dir
my-feature ~/proj/wt/my-feature ② hook, unprovisioned
pi-agent-dashboard ~/proj/pi-agent-dashboard ③ configured, no hook
— no initialize control —

✓ Set up project (indigo, ✦) = scaffold — visually distinct from ✓ amber Initialize = run repo hook. ✓ ③ shows nothing — already a project.

What each state resolves to

Driven by GET /api/git/worktree/init-status. New field: configured (present only when hasHook:false).

Stateinit-statusBeforeAfterComponent
① unconfigured { hasHook:false,
  configured:false }		 amber “Initialize” indigo “Set up project” ProjectInitButton
② hook, needs run { hasHook:true,
  needsInit:true }		 amber “Initialize” amber “Initialize” (unchanged) WorktreeInitButton
③ configured, no hook { hasHook:false,
  configured:true }		 amber “Initialize” (wrong) nothing —

Fail-open unchanged: a probe error → { hasHook:false } with configured absent → no button (scaffold requires strict configured === false).
