Same folder-action-bar row. Goal: kill the inline wall of monospace; keep progress + failure clarity. Terminal output becomes opt-in.

Problem: the running tail (and the failure stderr) render as a full <pre> inside the sidebar row — a scrolling wall of monospace that dominates the panel and reads like a terminal, not a UI.

A · Status chip + collapsed log — recommended

failure — compact, actionable, log opt-in

Why this works: default state is one friendly line. The last log line gives live progress without the wall. Full log is one click away (<details>, collapsed). Failure leads with a plain-language summary + Retry, not raw stderr.

Execution lives in a corner toast / slide-out, not the row. Row just shows a spinner pill. Cleanest sidebar, but the detail is detached from the folder it belongs to — and needs new toast plumbing.

D · Spawn / auto-init path — today: 100% silent

Current gap: when you spawn a worktree with auto-init ON, maybeAutoInitWorktreeOnSpawn fires the hook but discards its requestId — nothing subscribes, so init runs (and can fail) with zero feedback. cwd-keying lets the same toast/card pick it up automatically.

Feedback lives on the nascent session card — exactly where the user is looking after spawn. No floating element, no sidebar-row damage. Failure is now visible + retryable instead of a silently-broken worktree.

D2 · same corner toast as C, auto-triggered by spawn
my-featureos/add-searchspawning…
Auto-initializing my-feature 8s
$ pnpm install · resolving dependencies…

Reuses C's toast unchanged — spawn just registers the run by cwd and the toast appears. Detached from the card, but one surface for every trigger.

E1 · D1 success flash — the running → done → settled transition

On done, the chip flips to a green “✓ Initialized” for ~2s so you see it worked, then the init sub-state collapses away and the card settles to a normal ready session. Silent-vanish would leave you unsure init ever ran.

t+8s · running

Failure differs: never auto-collapses — the “✕ Auto-init failed · Retry” sub-state is sticky until dismissed or retried (losing a failure to a 2s timeout would be the worst case).

E2 · Concurrent inits stack — spawn several worktrees at once

cwd is the key, so N runs = N registry entries. Corner surface collapses them into one stack with a summary header; each row is an independent cwd (one still running, one done-fading, one failed-sticky). The per-card D1 sub-state renders in parallel on each session card regardless.

Initializing 3 worktrees 1 done · 1 failed
add-search $ pnpm install… 8s
✓ fix-auth Initialized · fading
✕ refactor-db exit 1 · pnpm install

Header summarizes at a glance; the stack auto-shrinks as runs finish. When the last run settles (all done), the whole stack fades — but any failed row keeps the stack open until cleared. Caps at ~4 visible rows with an +N more overflow.

Trade-offs

A best progress feedback · moderate footprint · recommended
B tightest · least feedback · trivial to build
C pristine row · detached context · new toast plumbing
D1 spawn feedback ON the session card · contextual · needs cwd-keyed runs
D2 spawn feedback via C's toast · one surface all triggers · detached
