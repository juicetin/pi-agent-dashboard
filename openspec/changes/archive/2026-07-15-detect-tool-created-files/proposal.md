# Detect tool-created files in the changed-file list

## Why

The changed-file list (`extractFileChanges` in `packages/server/src/session-diff.ts`,
surfaced by `GET /api/session-diff`) is built from **only** `Write` and `Edit`
`tool_execution_start` events (`WRITE_EDIT_TOOLS = {"write","edit"}`). Git enrichment
(`enrichWithGitDiff`) only *decorates* files already in that list — it never *adds* files.

So any file produced by a different action is invisible in the Files panel:

- a converter run (`pandoc … -o out.docx`, the `document-converter` skill),
- an image generated by `nano-banana` (`npx @the-focus-ai/nano-banana … --output logo.png`),
- a script writing output (`python build.py > report.json`, `tee`, redirects).

None of these are registered pi tools — they all run **through Bash** — so no Write/Edit
event carries their path, and the file never appears even though git already sees it as
new/untracked on disk. Users lose track of artifacts the agent just made.

## What Changes

1. **git-status becomes a file *detector*, not just an enricher.** In a git repo, the server
   runs `git status --porcelain` (with `cwd = session.cwd`), and unions each new/modified/
   untracked path into the changed-file list. Every porcelain path is C-unquoted, rename lines
   (`R old -> new`) resolve to the new path, and each is passed through the **same**
   `normalizePath(abs, cwd)` used for Write/Edit — guaranteeing one shared key space (no
   double-listing) and the same cwd containment. This is the reliable ground truth for "what
   changed on disk", independent of which tool did it.

2. **Bash commands become an *attributor* that only labels detected files.** The server scans
   `tool_execution_start` events with `toolName === "bash"` for output tokens
   (`>`, `>>`, `-o <p>`, `--output[= ]<p>`, `tee <p>`), building a path → command map. A
   detected file is *labeled* with the Bash call that most likely produced it. Inside cwd the
   attributor **never adds a file** — it only labels files the detector already found, which
   kills `grep -o pattern` / `ssh -o …` false positives (a named path that isn't detected gets
   no label). The stored label is redacted (secret shapes stripped) and length-capped.

3. **Each file carries its origin (file-level, not per-event).** `FileDiffEntry` gains
   `origin` (`write | edit | tool | mixed`), `producedBy` (redacted command), `detectedVia`
   (`git-status | bash-artifact`), and a reserved `previewable` flag. `FileChangeEvent.type`
   gains `"tool"`. Files with both a Write/Edit event AND on-disk detection are `mixed` — their
   real edits are kept and **no synthetic change event is injected** (no ghost row, no count
   inflation). The Files panel badges the row and shows `created by <command>` when attributed.

4. **Non-git sessions are covered, anchored to cwd.** When git is unavailable, the detector is
   the Bash-token scan + **in-cwd** `existsSync` (each parsed path is `normalizePath`'d to be
   cwd-contained before the existence check — no arbitrary-path probe). Best-effort by
   construction.

5. **Binary + size safety.** Before generating a synthetic new-file diff for a tool-detected
   file, the server binary-sniffs (NUL byte / known-binary extension) and enforces a size cap.
   Binary/oversized rows are listed with `origin: "tool"` and no text `gitDiff` (rendered via
   the existing image/preview dispatch). This fixes the headline case: a generated PNG no longer
   becomes a corrupt utf-8 "diff".

6. **Session-ownership gating (git state is cwd-scoped, not session-scoped).** Because
   `git status` reflects the shared working tree (multiple sessions can share a cwd), a raw
   union would surface files another session / a manual edit / a build touched. Each
   git-detected file is classified by ownership evidence from THIS session: a Write/Edit event,
   a Bash output-token match, or its mtime falling inside one of this session's Bash execution
   windows. Owned files (`sessionOwned: true`) go in `data.files`; the rest go in a separate
   `data.otherChanges[]`. The Files panel shows owned files normally and puts the rest under a
   muted, **collapsed** `▸ N other working-tree changes` group, with a "this session only"
   toggle that hides the group. Worktree-isolated sessions have an empty `otherChanges` (no-op).

## Non-Goals

- **Files created OUTSIDE the session cwd.** Deferred to a follow-up change. Listing arbitrary
  out-of-cwd paths introduces an unanchored `existsSync` probe oracle and a not-previewable UX
  gap (the `/api/session-file` 403 gate stays). v1 filters out-of-cwd exactly as today; the
  converter/image/script cases are covered whenever they write into the project.
- **Deleted files.** The feature is about tool-*created* files; deletions have no content to
  preview and are out of scope for v1.
- **No new file-producing pi tools.** We read the Bash events converter/image CLIs already emit.
- **No robust general shell parsing.** The Bash-token scan is a bounded heuristic (redirects +
  common `-o/--output/tee` flags); pipelines, subshells, `xargs`, env-var paths, and exotic
  quoting are explicitly out of guaranteed scope. Attribution is best-effort and, inside cwd,
  can only label — never fabricate — a file.
- **No change to the `/api/session-file` cwd 403 gate**, and no change to git diff / numstat
  enrichment semantics for files already listed.

## Discipline Skills

- **security-hardening** — parsing untrusted Bash command strings, the secret-redaction of
  `producedBy`, and keeping the non-git existence probe anchored to cwd (no probe oracle) all
  sit on a security boundary; the `/api/session-file` 403 gate must not regress.
- **systematic-debugging** — attribution correctness (right file ↔ right command, no
  false-positive re-tag of real edits) and the porcelain-key ↔ Write/Edit-key equivalence are
  evidence-driven; missed-detection and false-attribution are the failure modes to reproduce
  first.
