# Open P/D/S/T artifacts in a modal dialog on non-mobile viewports

## Why

Clicking an OpenSpec artifact badge (**P**roposal / **D**esign / **S**pecs /
**T**asks) today navigates to a URL-driven full-page overlay route
(`overlay-url-routing`, `fix-openspec-artifact-tab-url-sync`). The full-page swap
yanks the user out of their current view (session card, board, header) to read
one short artifact.

For a peek-and-dismiss action a modal is the better UX. On mobile a full-screen
route is still correct (a modal is cramped), so the dialog is gated to
**non-mobile** viewports; mobile keeps the existing route.

## What Changes

- **Non-mobile:** a new App-level gate opens a `<Dialog size="full" flush>`
  holding a new `OpenSpecArtifactDialog` reader body. P/D/S/T tab switching is
  **local state** — no URL push. Esc / backdrop / the reader's back control
  closes it. The current view stays mounted behind the dialog and is revealed on
  close.
- **Mobile:** unchanged — `navigate(buildOpenSpecPreviewUrl(...))` full-page
  route, browser Back closes.
- **Gate location:** a new `openArtifact(cwd, changeName, artifactId)` handler in
  `App.tsx` branches on the existing `useMobile()`. `useOpenSpecActions`
  stays navigate-only — the viewport gate lives where `isMobile` and the dialog
  state already live. All **5** badge wiring sites (one bare `onReadArtifact={handleReadArtifact}`
  reference + four cwd-closures) call `openArtifact` instead; `openArtifact` keeps
  the 3-arg signature so the bare-reference site swaps by reference.

### Viewport gate is not literally "desktop"

`useMobile()` returns true for width < 768px **OR** height < 600px. A short
desktop window therefore takes the mobile route. "Non-mobile" throughout means
"`useMobile()` is false", not "wide screen".

### Reader body: new component, not OpenSpecPreview reuse

`OpenSpecPreview` hardcodes `navigate()` in its `onTabChange`, so it cannot drive
local-state tabs. `OpenSpecArtifactDialog` instead mirrors the existing
`ArchiveArtifactReader` pattern (local `activeTab` → `useOpenSpecReader` →
`onTabChange={setActiveTab}`). This is a third copy of that ~15-line reader-body
pattern; extracting a shared body is deliberately deferred (would touch the
out-of-scope archive reader and route-match sites). Documented DRY debt. The
reader body must be wrapped in a height-constrained `flex flex-col` container
(the flush `Dialog` is not flex; `MarkdownPreviewView` uses `flex-1`), and must
render an explicit not-found state when the change is absent after load.

### Accepted trades

- **Ephemeral dialog:** the click-opened dialog is not deep-linkable and does not
  survive reload (pure local state — the chosen simpler model). The URL route
  still exists and still renders full-page on any viewport, so a **held** URL
  (bookmark, shared link, browser Back onto it) keeps working — deep-linking is
  not generated by a click but is not fully lost.
- **Dual display path on non-mobile:** click → dialog; direct URL → full-page
  overlay. Documented, intentional.

### Out of scope

- Archive browser (`ArchiveBrowserView` → its own local `handleReadArtifact` +
  `ArchiveArtifactReader`, `archive=true`) — untouched. Its isolation is a hidden
  coupling: routing archived badges through the main `openArtifact` would open
  the non-archive path (`useOpenSpecReader` `archive` defaults false) → "not
  found". Do not wire archived badges through `openArtifact`.
- Mobile behavior, `buildOpenSpecPreviewUrl`, the three route-match sites —
  unchanged.
- Pre-existing latent issues left as-is: `useOpenSpecReader`'s `AbortController`
  signal is never passed to `fetch` (stale writes suppressed by a guard, not by
  cancellation); `useOpenSpecActions` carries an unused `openspecMap` dep.
