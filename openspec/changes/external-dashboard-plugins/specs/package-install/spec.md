## ADDED Requirements

### Requirement: Successful global-scope package operations invalidate the plugin discovery cache

After a successful `POST /api/packages/install`, `POST /api/packages/remove`, or `POST /api/packages/update` operation with `scope: "global"`, the route handler SHALL invoke `clearDiscoveryCache()` from `@blackbelt-technology/dashboard-plugin-runtime/server`, then re-run `discoverPlugins()` and `loadServerEntries()` to register newly added plugins. The route handler SHALL then broadcast `plugins_changed { plugins }` with the refreshed status snapshot.

The handler SHALL perform this work AFTER pi's `installAndPersist` (or equivalent) resolves successfully, so a partial install does not leave the dashboard with a half-cleared cache. The cache invalidation SHALL run on every successful global operation regardless of whether the package being installed declares a `pi-dashboard-plugin` manifest — non-plugin pi extensions simply produce no new plugin entries.

#### Scenario: Plugin install triggers discovery refresh and broadcast

- **WHEN** the user installs `npm:pi-memory-honcho` (which declares a `pi-dashboard-plugin` manifest) globally
- **THEN** the route handler SHALL call `clearDiscoveryCache()`, re-run discovery, register any new server entries, and broadcast `plugins_changed` containing the new plugin's status to every subscribed browser.

#### Scenario: Pi-extension install (no plugin manifest) still invalidates cache

- **WHEN** the user installs `npm:pi-some-extension` globally and that package has no `pi-dashboard-plugin` manifest
- **THEN** the route handler SHALL still invoke `clearDiscoveryCache()` and `discoverPlugins()` (cheap because it is cached), and SHALL still broadcast `plugins_changed`; the snapshot SHALL be unchanged.

#### Scenario: Local-scope operations do NOT invalidate the cache

- **WHEN** the user installs a package with `scope: "local", cwd: "/proj/A"`
- **THEN** the route handler SHALL NOT call `clearDiscoveryCache()`; the plugin discovery cache pertains to global-scope plugins only in this release.

#### Scenario: Failed install does not invalidate cache

- **WHEN** `pi.installAndPersist` rejects with an error
- **THEN** the route handler SHALL NOT call `clearDiscoveryCache()` and SHALL NOT broadcast `plugins_changed`; the existing discovery state remains.

### Requirement: Plugin update reports `restartRequired` when an existing plugin's code changes

When `POST /api/packages/update` succeeds for a plugin previously loaded by the dashboard (i.e. its `id` already appears in the discovery cache before the update), the route handler SHALL set `restartRequired: true` on that plugin's `PluginStatus` entry in the next broadcast.

The dashboard React client registry is built at Vite time and cannot hot-swap a plugin's client bundle in a running process. The dashboard SHALL NOT auto-restart in this change; the user-facing UI in the Settings → General Plugins section is responsible for surfacing the hint.

#### Scenario: Updating an existing plugin marks it restart-required

- **WHEN** plugin `honcho` is loaded with `source: "global"` and the user invokes `POST /api/packages/update` against `npm:pi-memory-honcho`
- **THEN** the post-update `plugins_changed` broadcast SHALL contain the plugin's entry with `restartRequired: true`.

#### Scenario: Newly added plugin (not previously loaded) does NOT report restart-required

- **WHEN** the user installs a plugin for the first time
- **THEN** the post-install entry SHALL have `restartRequired: false` (or absent), because the new server entry was just registered and its client bundle was generated by Vite at the most recent build.
