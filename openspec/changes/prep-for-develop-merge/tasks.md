# Tasks

**Execution protocol:**

1. **No auto-commits â€” I write a commit script, you run it.** When a phase is complete and ready to commit, I will:
   - Write a shell script at `B:/tmp/pi-commit-<N>-<slug>.sh` (on this system, the `Write` tool resolves paths starting with `/tmp/` to `B:/tmp/` because the agent's cwd is on the B: drive â€” so I use the explicit drive-letter form to avoid confusion).
   - The script makes **multiple small commits** when a phase introduces more than one logically-distinct artifact. Each commit gets a **one-liner** conventional-commits message â€” no commit-body paragraphs, no bullet lists. Format:
     ```
     <type>(<scope>): <one-short-sentence>
     ```
     where `<type>` is one of `feat`, `fix`, `refactor`, `test`, `chore`, `docs`.
   - Prefer splitting a phase into 2â€“4 one-line-message commits over writing one long-body commit. E.g., Phase 0 produces three commits (one for each planning doc + one for the proposal), not one mega-commit.
   - Use paths rooted at the repo (`cd B:/Dev/BB/pi-agent-dashboard` at the top, then relative paths for `git add`).
   - Use `set -e` so the script aborts if `git add` or `git commit` fails.
   - The script SHALL be **idempotent**: if the script is being re-run (detected by matching the last commit's subject against expected), it first calls `git reset --mixed HEAD~N` to un-stage the old commits (keeps working tree on disk) before applying the new version. This lets you iterate on the script without manual revert.
   - Print the invocation on a single line in the form (git-bash-friendly):
     ```
     bash B:/tmp/pi-commit-<N>-<slug>.sh
     ```
   - Wait for your "continue" signal before starting the next phase.

2. **No auto-pushes.** Nothing in this plan pushes to `origin`. You open the PR from `windows-integration` â†’ `develop` yourself when ready.

3. **No full test suite.** `npm test` is unreliable in this repo. I will run *targeted* vitest invocations against specific test files only. When verifying a phase, I will run exactly the test files that phase touches.
   ```
   Example targeted test:
   npx vitest run packages/shared/src/__tests__/platform-paths.test.ts
   ```

4. **Stashes are already handled.** The ~640 LOC of uncommitted preload-fastify work (which you referred to as "stashed changes") is discarded in Phase 0.1a via `git checkout HEAD -- <files>` and `rm`.

5. **Commit-script cleanup.** After you confirm a commit succeeded, I delete its script (`rm B:/tmp/pi-commit-<N>-<slug>.sh`) before proposing the next one. This keeps the temp dir clean and prevents accidental reuse. If you want to keep them for audit, tell me at the start.

**Commit-script template I use for every COMMIT step (idempotent, multi-commit, one-line messages):**
```bash
#!/usr/bin/env bash
# Auto-generated by prep-for-develop-merge tasks.md phase <N>
# Idempotent: detects and undoes prior run of this phase if present,
# then applies the intended small commits.
set -e
cd B:/Dev/BB/pi-agent-dashboard

# Detect prior run by matching the LAST commit this phase makes.
LAST_SUBJECT="<type>(<scope>): <one-short-sentence> (last commit of the phase)"
PHASE_COMMIT_COUNT=<number of commits this phase makes>

if git log -1 --format=%s | grep -qxF "$LAST_SUBJECT"; then
  echo "â†¶ Prior run of phase <N> detected. Un-staging $PHASE_COMMIT_COUNT commits..."
  git reset --mixed HEAD~$PHASE_COMMIT_COUNT
fi

echo "â†’ Commit 1/N: <slug>"
git add <file1>
git commit -m "<type>(<scope>): <one-short-sentence>"

echo "â†’ Commit 2/N: <slug>"
git add <file2>
git commit -m "<type>(<scope>): <one-short-sentence>"

echo "âś“ Phase <N> complete. Commits:"
git log -${PHASE_COMMIT_COUNT} --format="  %h %s"
```

**Commit message style â€” one-line only. NO body paragraphs, NO bullet lists in messages. Examples:**
- `fix(windows): restore detach:false for pi-session spawn`
- `feat(server): add node-guard refuse-to-start`
- `refactor(platform): merge exec + subprocess-adapter + detached-spawn + spawn-mechanism into spawn.ts`
- `docs: add branch-comparison analysis`
- `chore(openspec): archive consolidate-platform-handlers`

Implementation order is deliberately sequenced so each phase is independently revertable and each phase's targeted tests pass before the next begins. Do not reorder without rereading `design.md` rationale.
