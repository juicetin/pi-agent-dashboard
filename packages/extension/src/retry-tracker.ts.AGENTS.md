# retry-tracker.ts — index

Pure helper class `RetryTracker` synthesizes `auto_retry_start` / `auto_retry_end` by OBSERVING pi's own retry behavior — NO regex. pi ExtensionAPI does NOT expose `auto_retry_*` (verified pi 0.70/0.73; https://github.com/badlogic/pi-mono/discussions/2073). Observe model: error `message_end` records a pending failure + emits NOTHING; `observeMessageStart` on the next assistant `message_start` (same turn, no user prompt) emits `auto_retry_start{attempt,maxAttempts:-1,delayMs:-1,errorMessage}`; non-error `message_end` closing a chain emits `auto_retry_end{success:true}`; terminal error `agent_end` in a chain emits `auto_retry_end{success:false,finalError}`. A non-retryable terminal error (pi never re-attempts → no fresh `message_start`) emits NO `auto_retry_start`. `RETRYABLE_PATTERN` deleted. Bridge drives `observeMessageEnd` (sync, before `agent_end`) + `observeMessageStart` (assistant) + `observeAgentEnd` + `noteAbort`. See change: simplify-error-retry-single-card.
